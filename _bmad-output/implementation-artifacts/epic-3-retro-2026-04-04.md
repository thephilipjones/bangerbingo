## Struggles & Lessons Learned

### `@hono/node-server` v1.19 doesn't export a `/ws` WebSocket helper
- **What happened (3.2):** Story assumed `createNodeWebSocket` from `@hono/node-server/ws` would be available. It isn't in v1.19. Had to use the `ws` npm package directly with `WebSocketServer({ noServer: true })` and the HTTP `upgrade` event.
- **Lesson:** Don't assume library sub-path exports exist without checking the installed version. For WebSocket work, the raw `ws` package + `noServer: true` pattern is more portable anyway — consider this the canonical approach for this project.

### Room code was 4 chars in code, 5 chars in spec
- **What happened (3.3):** AC 5 and the error message said "5 letters" but `generateRoomCode()` in `rooms.ts` produces 4 characters. Story 3.3 caught and corrected this during implementation (AC text, validation regex, sanitize slice, maxlength, error message all updated to 4).
- **Lesson:** When a client story consumes a server contract established in a prior story, check the implementation directly — don't rely on the spec text. The spec was wrong; the code was right.

### Double WebSocket lifecycle bug (JoinPage → RoomPage)
- **What happened (3.5):** `JoinPage.svelte` never closed `activeWs` on unmount. When `RoomPage` mounted, it called `connectAsGuest` again with the same name/code. The server closed the new socket with 4009 ("name taken") and the guest froze with no UI feedback.
- **Fix:** Close `activeWs` in `JoinPage.svelte` on unmount; pass the existing WebSocket (or name+code) to `RoomPage` instead of reconnecting.
- **Lesson:** When navigating between pages that share a WebSocket, treat the socket as owned by one page at a time. The handoff must be explicit — either pass the socket as a prop or close-and-reopen. Document this pattern for game round transitions in Epic 5.

### Spurious `host:reconnected` on first connect
- **What happened (3.5):** `isReconnect` was `true` when guests joined before the host had ever connected (room entry created by guest path, `room.host === null`). The server broadcasted `host:reconnected` to guests who never received `host:disconnected`.
- **Fix:** Added `hostHasEverConnected: boolean` to `RoomState` — only broadcast `host:reconnected` if this flag is set.
- **Lesson:** "Was previously connected" and "slot is currently null" are different conditions. Track them separately. Any stateful reconnect logic needs an "ever connected" flag.

### `ws` package test client timing
- **What happened (3.5):** A `delay(20)` wall-clock wait in a reconnect test was flagged as potentially flaky on slow CI.
- **Lesson:** In WebSocket integration tests, use structural assertions (wait for a specific message, use a Promise) instead of arbitrary `setTimeout` delays. The pattern is already established for `session:connect` — apply it consistently.

## Review Feedback Patterns

Across 5 stories, **~20 patches applied** and **~30 items deferred**. Dominant patch themes:

1. **Missing `onerror`/`onclose` handlers on WebSocket connections** (3.2, 3.4) — silent WS failures leave UI in stale state. Pattern: every WebSocket connection must handle both `onerror` and `onclose`.
2. **Socket/timer cleanup on component destroy** (3.2, 3.4) — WebSocket connections and `setInterval`/`setTimeout` calls not cleaned up in `onDestroy`. Pattern: every resource acquired in `onMount` must be released in `onDestroy`.
3. **Old socket not closed before slot overwrite** (3.2) — host reconnect overwrites the slot reference, leaking the old TCP connection. Pattern: always close the old socket before assigning a new one.
4. **Input normalization** (3.2) — whitespace-only guest names (`?name=%20%20`) passed the `!name` guard. Pattern: trim/normalize all user-supplied strings before storing or broadcasting.
5. **`broadcast()` must be fault-tolerant** (3.2) — one failing socket was aborting delivery to all remaining sockets. Pattern: wrap each individual `socket.send()` in try/catch within broadcast loops.
