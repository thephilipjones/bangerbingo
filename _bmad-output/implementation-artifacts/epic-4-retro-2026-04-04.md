## Epic Summary

Delivered the full card-generation pipeline: Spotify playlist/search API proxied through the server (4.1), the host round configuration screen (4.2), and deterministic per-player 5×5 bingo card generation with WebSocket broadcast and down-ranking of previously-played tracks (4.3). The server can now start a real game round end-to-end.

---

## What Went Well

- **Pure function extraction in `cards.ts` paid off immediately.** `buildPool`, `generateCard`, and `generateCards` are self-contained with no I/O, making them trivially unit-testable. The Epic 3 pattern of pulling logic out of handlers/components held up well.
- **Down-ranking worked cleanly.** The `buildPool` pool-ordering approach (push played songs to the back instead of excluding them) kept the card-gen logic simple and handled the edge case of pools smaller than 25 naturally.
- **`pendingRound` as a lightweight bridge between stories.** Storing the pending round config in-memory as a placeholder in 4.2, then consuming it in 4.3, kept each story independently shippable without prematurely committing to a schema.
- **Late-join handling was a clean add.** The `currentRound` snapshot on `RoomState` meant the `session:connect` path only needed a small branch — no new data structures.
- **Test count growth was proportional.** +45 tests across 3 stories; each story added integration coverage of the actual server behaviour, not just unit padding.

## Struggles & Lessons Learned

### Inline token refresh duplicated across handlers
- **What happened (4.1 → 4.3):** The inline token refresh block (check expiry, call `refreshWithRetry`, re-read from DB) was copy-pasted verbatim into `/music/search`, `/music/tracks/:playlistId`, and `POST /api/rooms/:code/round`. Three copies; one future bug surface.
- **Lesson:** By the second duplication, extract a `getValidAccessToken(hostId)` helper. This is the clear next cleanup before Epic 5 adds more Spotify calls.

### `playlistId` path param reaches Spotify unsanitized
- **What happened (4.1 review):** The path param is interpolated directly into the Spotify URL — an authenticated host can reach arbitrary Spotify API paths. Flagged in code review and deferred.
- **Lesson:** Any user-supplied ID passed to an external API must be validated against a known-safe format (UUID pattern for Spotify playlist IDs) at the router boundary. Should be patched before the app has more than one real user.

### `roomSockets` still never pruned
- **What happened:** This was flagged in Epic 3 (3.2) and again in Epic 4 (4.2 notes reference it). In-memory room state grows unbounded across server lifetime.
- **Lesson:** This is now a two-epic-old carry-over. It won't bite at ≤5 users, but it should be budgeted into Epic 6 hardening rather than continuing to defer.

## Review Feedback Patterns

Across 3 stories, **~8 patches applied** and **~15 items deferred**. Dominant deferred themes:

1. **Token refresh duplication** — three handler copies of the same inline block. Extract before adding more Spotify calls in Epic 5.
2. **Missing FK on `played_songs`** — `room_id` has no FK reference to `rooms(code)`; orphaned rows accumulate. Address in Epic 6 schema hardening.
3. **Non-deterministic test using `Math.random()`** — `generateCards` uniqueness test could theoretically flake on unseeded random. Seed the test or use a fixed pool.
4. **State mutation before persist in round start** — cards are broadcast before `recordPlayedSongs` writes to SQLite. Invert order if atomicity ever matters.

## Epic 5 Preparation Notes

Heading into the **Game Loop** epic, the key technical debt items to watch:

- **Extract `getValidAccessToken(hostId)`** before adding any more Spotify calls — the duplication is already at 3 copies.
- **WS page-handoff pattern** (established in Epic 3, 3.5) must be applied when transitioning from the lobby to the active game round — the socket is owned by one page at a time.
- **`state_changed` burst on seek** (noted in Epic 2 spike) — debounce or skip events until playback position advances past the seek point before trusting state. This is the most likely source of game loop bugs.
- **Host reconnect mid-round** receives no `round:start` re-send — need equivalent late-join logic on the host reconnect path in Epic 5.
- **Blank card for late-join guest in round 1** — never backfilled when round 2 starts. Revisit when multi-round UX is designed.
