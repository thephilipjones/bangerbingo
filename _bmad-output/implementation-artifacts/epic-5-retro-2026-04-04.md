## What Went Well

### `bingo.ts` pure function extraction
All game state transitions (`initTiles`, `applyMask`, `startReveal`, `finishReveal`, `toggleMark`, `applyWinPath`) are pure functions with no I/O. 20 unit tests, no component harness needed. The discipline to push logic out of `RoomPage.svelte` into a standalone module paid off immediately — `applyWinPath` was already available when 5-5 needed it, with no interface changes required.

### `songHistory` design threaded three stories without modification
Defined in 5-1 as an append-only field on `RoundState`, consumed directly in 5-5 (win validation checks `songHistory` for claimed tiles) and 5-6 (history drawer, late-join sync, host reconnect re-send). No renegotiation of the interface mid-epic. Upfront data modeling was correct.

### Epic 2 spike findings were actually applied
`seekPositionMs: 60_000`, the `auth_error` double-fire quirk, and the `state_changed` burst at seek position were all documented in the throwaway spike and correctly pre-integrated into 5-1 and 5-4 without re-discovery cost.

### Zero regressions across all 6 stories
168 → 186 → 208 → 212 → 218 → 224 → 231. Every story increment left the full suite green. No story broke prior work.

### Story handoff notes were surgical
Each story's Dev Notes identified exact file paths, line numbers, and export shapes the next story would consume. Reduced per-story context-building overhead significantly.

### Popup re-auth flow was architecturally clean
Threading `authEvents` from `refresh.ts` through `ws.ts` to the client without a new polling endpoint — using the existing EventEmitter and WS connection — was the right approach. 5-6 resolved the auth re-auth problem without adding complexity to the server.

## Struggles & Lessons Learned

### Timer management was the consistent blind spot

- **5-1 had 7 review patches** — the most of any story. Key bugs: duplicate `songHistory` entry on pause-resume (would have silently corrupted win validation in 5-5), stale auto-advance timer closure capturing `roomState` by reference (could have advanced a new round when old timer fired), missing pre-game pause guard (`/pause` with `currentSongIndex === -1` would broadcast `song:pause { songIndex: -1 }`).
- **Lesson:** Timer management requires explicit reasoning about object lifetime and closure capture. Write the `clearTimeout` and guard clauses before the `setTimeout`, not after.

### Lifecycle cleanup was a recurring miss

- 5-2: `setTimeout` for `finishReveal` not cancelled in `onDestroy` — leaks on component unmount.
- 5-3: `undoTimer` not cancelled in `onDestroy` — could fire an orphaned `round/end` POST after unmount.
- Same category of miss, two stories apart. Not caught proactively.
- **Lesson:** Every `setTimeout`/`setInterval` in a Svelte component needs a paired `onDestroy` cancellation. Treat it as a checklist item before marking a story ready for review.

### Race conditions on shared state appeared in every story touching async paths

- 5-5: `/round/claim` set `round.ended = true` *after* `await ctx.req.json()` — race window between parsing the body and locking the claim.
- 5-6: `reinitSdk()` had no concurrent-call guard — rapid `auth:restored` events could trigger repeated player disconnect/reinit loops.
- 5-4: `POST /sdk/device` silently discarded `deviceId` when `roomState` was absent — returned 200 but stored nothing.
- **Lesson:** Any mutation of shared in-memory state that follows an `await` needs an in-flight guard or an optimistic lock set *before* the await. Write the guard first.

### `getValidAccessToken` extraction was flagged in Epic 4 retro but not done

- Epic 4 explicitly prepared: "Extract `getValidAccessToken(hostId)` before adding any more Spotify calls — the duplication is already at 3 copies."
- Epic 5 added more Spotify calls (5-4: `startSong`, `/round/pause`) without extracting the helper. There are now at minimum 3 inline copies.
- **Lesson:** Retro preparation items that say "before adding X" must be the first task in the first story that adds X. Not a separate story — the first task.

### Fire-and-forget Spotify API calls have no error visibility

- `startSong` and `/round/pause` in `rooms.ts` call the Spotify Web API with `.catch(() => {})`. Silent failure means no signal when audio doesn't start.
- Intentional MVP trade-off, but not acceptable in a deployed environment where Philip can't open the server console.
- **Lesson:** Fire-and-forget is acceptable for non-blocking calls; silent swallow is not. At minimum: `.catch((err) => console.error('[spotify]', err))`.

## Review Feedback Patterns

Across 6 stories: **~27 patches applied**, **~35 items deferred**, **~20 dismissed**.

Dominant patch themes:
1. **Timer lifecycle** — missing pre-game guards, stale closures, missing `onDestroy` cleanup (5-1, 5-2, 5-3)
2. **Race conditions on shared state** — optimistic lock placement, concurrent call guards (5-5, 5-6, 5-4)
3. **Stuck UI state** — `isClaiming` permanently stuck on unexpected status codes (5-5), `sdkReady` not reset on error handlers (5-4)
4. **Missing state resets on round transitions** — `winData` not cleared on `round:start` (5-5), `round.ended` not validated before claim (5-5)

Dominant deferred themes:
1. Accessibility (focus traps, keyboard handlers) — SongHistoryDrawer, HostControlsPanel confirm dialog
2. Mid-round reconnect doesn't replay current `song:start` state — deferred from 5-3
3. Fire-and-forget control buttons with no error handling — `handlePlayPause`/`handleNext` in HostControlsPanel
4. `round.active` stays true after `songs:exhausted` — misleading state, acceptable for MVP

## Epic 6 Preparation Notes

Epic 6 scope: **Deploy & Harden** — Docker Compose single-command deployment (FR41), env secrets (FR42), Spotify credentials at deploy time (FR43), account disconnect/reconnect (FR5).

### Pre-deploy blockers (must address in Epic 6 regardless of story assignment)

- **Session cookie signing** — `SESSION_SECRET` is validated at startup but never used. Wire actual signing before any internet-facing deployment. Unsigned cookie is trivially forgeable.
- **`playlistId` sanitization** — validate against Spotify playlist ID format at the `music/router.ts` router boundary before deploying with real users.
- **`getValidAccessToken(hostId)` extraction** — 3 inline copies. Epic 6 will add at least one more Spotify call (disconnect/reconnect for FR5). Extract the helper as the first task of whichever story touches `rooms.ts` or `music/router.ts` next.
- **Server Spotify call error logging** — add `.catch((err) => console.error('[spotify]', err))` to `startSong` and `/round/pause`. Silent failure is not acceptable in production.

### Suggested story structure

- **6-1: Container & Config** — Docker Compose, env secrets, Spotify credentials at deploy time, production `NODE_ENV` wiring (FR41-43)
- **6-2: Auth & Input Hardening** — session cookie signing, `playlistId` sanitization, `getValidAccessToken` extraction, Spotify call error logging
- **6-3: Memory & State Cleanup** — `roomSockets` pruning, `played_songs` FK, `refreshScheduler` test guard, `round.active` post-exhaustion state

FR5 (Spotify account disconnect/reconnect from settings) is lower priority for the friends-and-family use case — consider post-launch.

### Dependencies on Epic 5 work

Epic 6 inherits a clean game loop. All Epic 5 deliverables are stable. No Epic 5 rework is required before starting Epic 6 stories. The debt items above are additive hardening, not corrections.
