## Epic Summary

A three-story epic that reframes the host from "locked to the in-browser Web Playback SDK" to "a pure Spotify Connect remote that can route audio to any Connect-capable device." The delivery sequence was intentional and clean: server first, UI second, smart defaults third.

1. **10-1 — Device List API & Live-Swap Endpoint.** Server-only. Unified the existing `/sdk/device` handler into a shared `handleSetPlayerDevice` function registered on both `POST /api/rooms/:code/player/device` (canonical) and `POST /api/rooms/:code/sdk/device` (legacy alias). Added `GET /api/rooms/:code/player/devices` with full error-status coverage (401/403/404/503/502). Live mid-round transfer via `PUT /v1/me/player` — same-device no-op, paused/between-rounds store-only, active-playing transfer. OAuth scope bumped (`user-read-playback-state` + `user-modify-playback-state`) — existing hosts must re-login once. +23 tests; 440/440 suite green.
2. **10-2 — Device Chip + Picker UI.** Client-only. New `DeviceChip.svelte` (icon-only on mobile, icon+name on desktop) and `DevicePicker.svelte` (bottom-sheet on mobile, 360px popover on desktop). Mounted at `HostRoomPage` level for correct z-index isolation. Optimistic update + revert-on-failure + confirmation pill. AdvancedSettings "Playback Device" row in live mode. 17 patches from code review applied (race condition, pickerError lifecycle, iOS sticky hover, a11y aria attributes, AbortController on fetch, etc.). 440/440 suite unchanged (pure UI story).
3. **10-3 — SDK Default, Preference Persistence & Failure Path.** Client + hostPrefs. SDK `ready` callback now sets `selectedDevice = { id, name: 'Bangerbingo (this browser)', type: 'Computer' }` as the default when `selectedDevice === null`. `preferredDeviceId` added to `hostPrefs` schema (backward-compatible additive field, schema version stays at 1). Explicit picks persist; SDK-generated ids are never persisted (they change per session). `SdkFailureBanner` rewritten: old dead-end "Open in Spotify app" → new "Browser playback unavailable — pick a device" + "Pick a device" CTA. Banner auto-dismisses on successful pick. Empty-state 3-step onboarding copy when `sdkFailed`. `svelte-check` cleared to 0 errors (down from 8 pre-existing). +3 hostPrefs tests; 443/443 green.

## What Went Well

### Story sequencing created clean consumer contracts

10-1 was server-only with precise field-set documentation (`id`, `name`, `type`, `is_active`, `is_restricted`, `volume_percent` — no extras, no renames). 10-2's client code consumed exactly that contract. 10-3 extended `HostRoomPage` state without touching any of the endpoints. No "back-negotiate the API shape" moments; the sequence prevented mid-epic API drift that would have required coordinated rewrites across stories.

### Proactive scope risk was flagged and resolved inside the story it belonged to

10-1's "Latest Tech Information" section contained a prominent scope-risk callout: the OAuth scope list at `src/server/auth.ts:105` was missing `user-read-playback-state` and `user-modify-playback-state`. The dev agent was explicitly instructed to verify empirically before writing handler code. It did — confirmed the scopes were absent, bumped them in the same PR, and documented the re-login requirement for existing hosts. This is the correct pattern: surface the risk, verify the assumption, close it in the story where it matters. No separate "emergency patch" story needed.

### `persistRoomState` gap closed as an intentional bonus

The legacy `/sdk/device` handler had never called `persistRoomState`. This meant a server restart after SDK init but before any explicit device pick would lose the SDK device id. The unified handler in 10-1 calls `persistRoomState` unconditionally on every successful write (both paths), which closes this gap for free and makes 10-3's preference-persistence round-trip reliable from day one. The decision was documented in Dev Notes explicitly ("treat this as intentional") so future readers don't revert it.

### Component architecture decision was correct and held through all three stories

10-2's Dev Notes made an explicit call: `DevicePicker` mounts at `HostRoomPage` level, not inside `HostMiniPlayer` or `AdvancedSettings`. The z-index reasoning was documented (DevicePicker at 156 must sit above HostControlsOverlay at 150). This decision proved correct — no z-index surprises surfaced during review, and it meant 10-3 could thread `sdkFailed` to the picker as a single prop addition without any structural rework. One component, two entry points, one state owner.

### `pickerSource` tristate emerged naturally and cleanly

10-2 introduced `pickerSource: 'chip' | 'settings'` to suppress the confirmation pill for AdvancedSettings picks. 10-3 extended it to `'chip' | 'settings' | 'banner'` — the banner auto-dismiss is the feedback for a banner-origin pick, not the floating pill. The tristate was designed with the right semantics from the start (a source enum, not a boolean flag), so the extension was a one-line type widening with no logic changes. This is the kind of upfront design decision that pays forward.

### `preferredDeviceId` persistence semantics are tight

The three "do NOT persist" rules in 10-3 were all correct and correctly followed:
- Do NOT persist the SDK device id (it's session-scoped and changes on every `new Spotify.Player()` call).
- Do NOT persist on failed picks (only `res.ok` branch writes).
- Do NOT clear `preferredDeviceId` on device absence (the device may just be asleep; it should resolve on next load).

The result is a persistence model that "just works" across the common iOS host flow: desktop session → pick phone → phone goes offline → reload → falls back to SDK device → phone comes back → next reload prefers phone again. No explicit "refresh my preference" step required.

### Review feedback patterns improved across the epic

10-1 review: 4 patches, 4 deferred (clean server-side story with tight ACs). 10-2 review: 17 patches, 4 deferred (more surface area from UI complexity — appropriate). 10-3 review: 2 patches, 2 deferred (small focused wiring story — few surprises). The volume of review feedback tracked correctly with story complexity. No story had catastrophic review surprises.

### `svelte-check` cleared as a side effect

The 8 pre-existing `svelte-check` errors that had persisted since before Epic 10 were cleared in a housekeeping commit (`c35f7ee`) between 10-2 and 10-3. The Epic 10 codebase exits with 0 `svelte-check` errors. Not an Epic 10 deliverable per se, but a genuine quality improvement.

## Struggles & Lessons Learned

### 10-2 had too many HIGH-severity review findings for a spec-covered story

17 patches applied in 10-2, including two HIGH-severity issues: (1) race condition on rapid successive device picks — `prevDevice` captured per-call was stale if responses interleaved; (2) `pickerError` lifecycle broken — picker closes synchronously before POST resolves, so inline errors from the failing attempt were never rendered, and stale errors from prior failures surfaced on next open. Both are exactly the class of bug a spec-careful implementer should have caught pre-review: AC #6d explicitly describes the "stores an error string that is shown…the next time the picker opens" behaviour; AC #6 describes the optimistic close; the lifecycle conflict is deducible from those two ACs together.

10-2 was implemented with `claude-sonnet-4-6` while 10-1 and 10-3 used `claude-opus-4-7`. Whether model selection contributed to the difference in review-finding density is unclear — the UI story was also the most architecturally complex of the three. The pattern to watch: if a story's review regularly surfaces bugs that are deducible directly from the ACs, the pre-review implementation step is not reading the ACs carefully enough.

**Lesson:** For UI stories with optimistic-update + async-revert patterns, explicitly trace the "what if POST returns after picker is closed?" lifecycle in dev notes before coding. The race between async resolution and UI teardown is a recurring bug class in this codebase.

### The probe-then-flag pattern in 10-1 should be the standard for all API scope/contract risks

10-1's scope-risk callout worked because it was the first task listed in the execution instructions: "Before writing handler code, hit the endpoint and check the response." This created a hard gate. In earlier epics, similar risks (auth scope, endpoint contract changes) were sometimes discovered mid-implementation. The explicit "pause and verify" first-task instruction is the mechanism that made it work. It should be the template for any story that depends on an external API contract assumption.

**Lesson:** Whenever a story's server endpoints require OAuth scopes, API contract shapes, or third-party API behaviour assumptions, make "verify assumption X empirically" the first subtask in the task list, not a note buried in Latest Tech Information.

### `sdkDeviceId` field name is now a named misnomer in three story files

10-1 Dev Notes: *"Don't rename `sdkDeviceId` — it's a misnomer at this point but renaming is a separate refactor, do NOT rename in this story."* 10-2 and 10-3 echoed this. The field now stores any Spotify Connect device id (phone, Sonos, computer) but is still called `sdkDeviceId` in both server memory state and `persistRoomState` output. Three stories in, the misnomer is load-bearing enough that a rename requires a coordinated server-client-persistence update. The cost of deferring this has grown, not shrunk.

**Lesson:** When a field name that crossed a story boundary is documented as a misnomer in three consecutive story files, the rename cost has become more predictable (not less). If the field is touched in Epic 11, rename it there. Don't carry the documentation burden into Epic 12.

### Still not deployed to production

Epic 9 retro action item #2: "Prod deploy before Epic 10 dev begins." Not done. The two-epic prod deploy backlog from Epic 9 is now a three-epic backlog (Epics 8, 9, 10 all run on dev only). The multi-device feature (Epic 10) depends on the OAuth scope bump landing in production, which requires a prod deploy. Until it does, the feature is dev-only and existing prod hosts will get 403s from Spotify on `/player/devices`. This is now a near-blocking concern for anyone who tries to use the new picker on the prod install.

**Lesson:** The prod deploy can no longer be deferred to "whenever" — Epic 10's OAuth scope bump is not backward-compatible for prod hosts. The next thing that ships to prod should be the Epic 8 + 9 + 10 bundle. Frame it as "deploy now to stop the divergence from growing further," not "deploy when everything is polished."

## Key Takeaways

1. **Server-first sequencing with explicit field contracts prevents mid-epic API drift.** 10-1 specified exactly which Spotify response fields pass through (`id`, `name`, `type`, `is_active`, `is_restricted`, `volume_percent` — no extras). 10-2's client consumed that contract without negotiation. When the server story documents the response shape precisely, the client story can be written without reading the server implementation.

2. **Optimistic-update + async-revert patterns need explicit lifecycle tracing in dev notes, not just ACs.** 10-2's two HIGH-severity review bugs (race on rapid picks, broken pickerError lifecycle) were both deducible from the ACs but required tracing the "picker closes before POST resolves" timeline. ACs describe *what*; dev notes should trace the *timing*. Add a "lifecycle diagram" note for any story with an optimistic-close pattern.

3. **The "pause and verify" first-task instruction is the mechanism that makes risk callouts work.** 10-1's scope-risk callout worked because it was the first subtask, not a footnote. The same risk documented in a "Latest Tech Information" section that isn't gated by a first-task instruction can be skipped. For any API contract assumption, the first subtask should be "verify assumption X empirically — if fails, stop and flag before writing handler code."

4. **Persistence semantics for preference storage are worth explicit "do NOT" rules.** 10-3's three "do NOT persist" rules (no SDK id, no failed-pick id, no absence-triggered clear) prevented three common mistakes that would have corrupted the preference across sessions. These aren't obvious from the ACs alone; they require understanding the SDK's session-scoped id lifecycle. Future preference-storage work should carry the same explicit anti-patterns.

5. **Three consecutive epics off-prod is the hard line.** The prod deploy backlog started in Epic 8. The Epic 9 retro flagged it. It still hasn't happened. Epic 10 has made it load-bearing: the OAuth scope bump requires a prod deploy for existing hosts to use the new feature. "Ship when ready" is a valid stance; "ship never because the backlog keeps growing" is not. The next prod deploy is the most important non-story task in the project right now.
