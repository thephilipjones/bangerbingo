## Challenges

1. **12-2 was the heaviest story in the epic** ‚Äî 16 code-review patches. Root causes:
   - iOS platform complexity genuinely hard to anticipate (MacIntel UA detection, iPadOS 13+ quirk, concurrent-resume races)
   - `sdkReconnecting` state had multiple error paths not enumerated in spec
   - Story covered 5 distinct concerns across 13 ACs ‚Äî borderline for a split

2. **Manual on-device verification structurally incomplete** ‚Äî 12-2 journeys A‚ÄìG (iPhone + Bluetooth + Connect speaker) cannot be executed from CLI. Remain on Philip's plate.

3. **Pseudocode in ACs conflicted with real code** ‚Äî 12-3 and 12-2 both had spec deviations discovered at implementation time. Pseudocode in ACs is high-risk if not verified against live code shapes before publishing.

4. **iOS edge cases discovered in review rather than design** ‚Äî 4xxx terminal codes (12-1), MacIntel UA / nudge race (12-2) were found in review, not anticipated in story design.

5. **Client test surfaces remain shallow** ‚Äî Multiple deferred items across 12-3 and 12-4 acknowledge that client tests mirror handler logic in local copies rather than mounting Svelte components. The testing standard is pragmatic but accumulates risk.

---

## Key Insights

1. **Visibility + heartbeat catch different failure modes** ‚Äî keep both: heartbeat for foreground-network-drop, visibility for phone-lock/app-switch. They're cheap and complementary.

2. **Idempotency guards are correct in isolation but incomplete on reconnect** ‚Äî 12-3's core lesson. The server correctly said "already told you" while the client had lost the marks. The fix: separate unicast replay path targeting the returning socket only, not a change to the idempotency guard.

3. **Source-of-truth reconciliation > scattering drift correction** ‚Äî `/host/resume` covers all platforms; it's the pattern to extend. Don't scatter correction logic into individual event handlers.

4. **The `hasSeenRoundStart` guard (12-4 Track C) survived multiple reviews** ‚Äî An inverted flag silently killed casual mode on every reconnect/let-it-ride. Casual-mode state should have a dedicated test surface, not be buried in general state tests.
