## Epic Summary

Turned the working game into a deployable production system: server binds to `0.0.0.0` for Tailscale multi-device testing (6-1); multi-stage Dockerfile and `docker compose` with volume-mounted SQLite, `DB_PATH` injection, and `/healthz` endpoint (6-2); Caddy reverse proxy for HTTPS/WSS with auto-cert via Let's Encrypt (6-3); server-restart state recovery via an `active_rooms` SQLite table with `persistRoomState`/`rehydrateRooms` (6-4); host Spotify disconnect/reconnect via `/account` settings page, reusing the existing PKCE flow (6-5); Gitea Actions CI/CD with dual-stack staging + prod deploy, tag-based prod promotion, and a smoke test checklist (6-6).

## What Went Well

### Session cookie signing was clean and test-first

The unsigned-cookie debt from Epic 3 was cleared in 6-2. `signUserId()` / `verifyUserId()` in `auth.ts` using `SESSION_SECRET` with HMAC, and three test assertions covering valid-signature, invalid-signature, and unsigned-raw-user-id cases. Pre-deploy blocker closed properly.

### `callSpotify()` helper resolved the access_token duplication

Rather than a standalone `getValidAccessToken(hostId)`, a `callSpotify()` wrapper in `rooms.ts` centralises token fetch, 401 retry, 404 device-reactivation, and error logging in one place. The result is more comprehensive than the factored-function approach called for in the Epic 4/5/7 retros — every Spotify call now has consistent error visibility. The 3+ inline copies are gone.

### Error logging on Spotify calls is comprehensive

`console.warn` on missing access_token, `console.error` with body on 401, 404, non-2xx, and network error. Silent fire-and-forget is eliminated. Production failures will surface in logs.

### `playlistId` validation at router boundary

`/^[A-Za-z0-9]{20,30}$/` guard in `music/router.ts` before any Spotify API call. The Epic 4 / Epic 7 debt is closed.

### `roomSockets` pruning implemented

`roomSockets.delete(roomCode)` called on room close and `active_rooms` row deleted in the same path. In-memory leak closed after 4 epics.

### Restart recovery architecture was surgical

`active_rooms` table snapshots only at meaningful transitions (round start, each `song:start`, `round:win`, `round:end`). No per-tap pressure. `rehydrateRooms()` on boot with `host: null`, `guests: new Map()`, `paused: true` — correct defaults for a reconnection-first model. The existing 5-6 re-send logic handled host/guest card rehydration without modification.

### Dual-stack deploy on a single LXC

Distinct compose project names (`bb-staging` / `bb-prod`), distinct named volumes, and distinct env files. Two independent stacks running on one host with no shared state contamination.

## Struggles & Lessons Learned

### Post-story fix volume was the highest of any epic

Seven fix commits landed after the 6-6 feature commit — more than any prior epic. Categories:

1. **Healthcheck failures** (2 commits): IPv6 resolution (`127.0.0.1` vs `localhost`) and unquoted `$PORT` in the compose healthcheck shell command. Both were caused by gaps in local testing — the healthcheck was never exercised with a running container before merge.
2. **SSH port** (1 commit): Gitea Actions used port 22; the Proxmox LXC runs SSH on 2222. Not discoverable without a real deploy attempt.
3. **CI workflow / Ansible alignment** (1 commit): Generated workflows assumed a plain Docker host; the actual infra uses Ansible-managed SSH keys and a non-standard deploy path.
4. **Concurrency + tag format** (1 commit): `prod-*` tags could race if pushed in rapid succession; tag format validation was missing.
5. **SPA fallback** (1 commit): Direct navigation to `/account` (and other client routes) returned 404 from the production server — only discoverable in a container deploy, not local dev.

**Lesson:** Infrastructure stories need a real-deploy smoke test as part of the definition of done. "Does it build?" is not the same as "Does it work on the actual target?"

### Epic interruption created debt accumulation debt

The hold on 6-2+ (2026-04-05) while Epic 7 ran meant the Epic 5 retro action items aged to 3+ epics before resolution. The Epic 7 retro had to re-document them, adding overhead. The interruption was the right call (Epic 7 UX restructure de-risked the deploy), but the carry-forward cost was real.

### `handlePlayPause`/`handleNext` error swallowing: fourth consecutive deferral

Flagged in Epic 5, prepared in Epic 6 prep notes, deferred in Epic 7, and not addressed in Epic 6 6-2+. The host still has no feedback when playback controls fail silently. This is a production observability gap, not a polish item. It must not carry into Epic 8.
