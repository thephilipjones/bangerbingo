## Epic Summary

A single-story research spike to validate that the Spotify Web Playback SDK can seek to a track position and play a timed clip. Deliverable was a throwaway `spike-sdk.html` (gitignored). The spike succeeded in validating the core capability and surfaced several critical implementation details Epic 5 must handle.

---

## What Went Well

- **Spike scope was well-bounded.** No TypeScript, no Svelte, no Hono — pure HTML/JS. This kept the iteration loop fast and kept findings clearly attributable to the SDK rather than the app framework.
- **The correct seek pattern was discovered early.** `player.seek(ms)` alone has no effect before playback is active. The correct pattern is `position_ms` on the `/v1/me/player/play` Web API call. This would have been a painful discovery mid-Epic 5 game loop work.
- **iOS fallback path validated.** The `spotify:track:<id>` deep link approach for iOS was confirmed. The iOS disclaimer added in Story 1.1 is justified.
- **`state_changed` burst behavior documented.** The SDK fires `state_changed` 8–9 times in rapid succession when playback starts at a seek position. Epic 5 must debounce or skip events until position advances past the seek point — this is now captured in `deferred-work.md` before it can cause a surprise.

---

## Struggles & Lessons Learned

### `player.seek()` is not standalone
- **What happened:** Spike confirmed the SDK's `player.seek(ms)` method only works after playback is already active on the device. The spike pivoted to `position_ms` on the Web API `/play` endpoint, which works correctly on first call.
- **Lesson for Epic 5:** `SpotifySDKProvider.play(trackUri, seekMs)` should use `position_ms` in the initial `/play` call, not a post-play `player.seek()`. Document this in the Epic 5 story notes explicitly.

### Firefox / browser compatibility
- **What happened:** Initial init latency measurement of 1130ms was from macOS Safari (unsupported browser). Corrected to Firefox (731ms) and Brave/Chrome (490ms) after review.
- **Lesson:** Always validate SDK behavior in a supported browser (Chrome/Firefox desktop) before recording measurements. Safari is not supported by Spotify's Web Playback SDK.

---

## Review Feedback Patterns

This was a throwaway spike — patch-level issues (timer leaks, NaN inputs, unhandled rejections) were noted in the review but intentionally not fixed. All deferred with explicit notes to address in Epic 5's `SpotifySDKProvider`. This was the right call: investing in polish on throwaway code would have been waste.

However, two **decision-level findings** are worth carrying forward:

1. **`seek()` mid-clip never validated** — `player.seek(ms)` during active playback (e.g. jumping within a clip) was not tested. Epic 5 must decide whether to use it or rely solely on `position_ms`. Currently unknown if it works reliably.
2. **`state_changed` debounce required** — the burst of 8–9 events at seek position must be accounted for in the Epic 5 game loop event handler. Without debouncing, the host UI could flicker or miscount played songs.

## MusicProvider Interface Sketch (from Spike)

The spike validated this interface shape as the right abstraction for the SDK:

```ts
interface MusicProvider {
  play(trackUri: string, seekMs: number): Promise<void>
  pause(): Promise<void>
  resume(): Promise<void>
  next(): Promise<void>
  onReady(cb: (deviceId: string) => void): void
  onError(cb: (message: string) => void): void
}
```

Epic 5's `SpotifySDKProvider` should implement this interface. The `play(trackUri, seekMs)` signature encapsulates the `position_ms` pattern discovered in this spike.
