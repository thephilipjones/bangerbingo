## Epic Summary

Turned an otherwise competitive single-round game into a *relaxed* session format. Five stories landed back-to-back across 2026-04-14 and 2026-04-15:

1. **8-1 — Win Moment Hold & Audio Presets.** Removed the guest auto-dismiss timer on `WinOverlay`; added per-role Dismiss CTAs (immediate for the winner, 2 s delay for other guests, unchanged host CTAs); introduced the `AudioPreset` type (`hype` / `deadpan` / `minimal`) with a Vibe picker in `RoundConfigOverlay` and three visual variants in `WinOverlay`.
2. **8-2 — Session Statistics.** Added in-memory `SessionStats` (`winsByName`, `lastRoundWinner`) with a `stats:updated` WS event emitted after every `round:win` and after every no-winner `round:end`; Players List renders `×N` counts + `Last round ✓` pills, gated behind `highestRoundNumber >= 2` so one-off games stay clean; late joiners receive seed state on `session:connect`.
3. **8-3 — Continuous Mode.** Host toggle, `continuous:countdown-start|cancel` WS events, 10 s auto-start countdown driven from a new `POST /round/dismiss-win` host-authoritative dismiss, and a host mini-player button with countdown display. Extracted `startRound` into a reusable helper so the HTTP handler and `startContinuousRound` share one path.
4. **8-4 — Casual Mode Permission & Player Toggle.** Host-side `allowCasualMode` permission flag per round; player-side WS toggle (`player:casual-mode-changed`) with a `☕` indicator in the Players List; `casualModeNames` seeded through the existing late-join chain.
5. **8-5 — Casual Mode Auto-Mark Engine.** `runCasualModeSweep` helper fires on every track change and sweeps played history for each opted-in player; per-player `autoMarkedTileIndices` deduping; catch-up sweep on enable/reconnect with a "Caught up on X songs" toast; client auto-claim 600 ms after an auto-mark creates a winning line.

Also shipped on day one of the epic: **commit `6430ac4`** surfaced `handlePlayPause` / `handleNext` errors to the host and added the guest session-end banner — closing both Epic 6 action item #1 (four-epic deferral) and an Epic 7 deferred UX gap in a single 40-line change.

The epic was faster and smoother than any previous one. Post-story fix volume was minimal compared to Epic 6's seven-commit tail.

This is the first time in the project that a retro action item was resolved proactively instead of under duress.

### The `session:connect` seeding chain is now a reliable pattern

Stories 8-1 through 8-4 each extended the guest-join plumbing chain:
- 8-1: (`audioPreset` lives in `RoundConfig`, no late-join seed needed — reused pattern)
- 8-2: `winsByName`, `lastRoundWinner`
- 8-3: `continuousMode`, `countdownRemainingMs`
- 8-4: `casualModeNames`

The chain is now well-worn: `ws.ts session:connect payload → connectAsGuest → GuestHandlers.onConnect → JoinPage.onJoined → App.handleJoined → RoomPage initial* prop → createGameState({ initial* })`. Each new field added ~4 test updates and zero architectural debates. Story 8-4 Dev Notes cite 8-2/8-3 as the template without ambiguity. The late-join reconnect story is solid enough that no epic-8 reconnect bug survived into post-review fixes.

### `startRound` helper extraction (8-3) paid off immediately in 8-4 and 8-5

The 8-3 refactor hoisted round-start logic out of the HTTP handler so Continuous Mode's auto-start could share it. When 8-4 needed `playerCasualModes` to reset on every new round, the reset went inside `startRound` — not inside the HTTP handler — and automatically covered the continuous auto-start path. When 8-5 needed `autoMarkedTileIndices: new Map()` to initialise per round, same deal. Two downstream stories benefited from a refactor they didn't ask for. This is the pattern Epic 6's retro called out as the right instinct ("consolidate when the second caller arrives, not on spec").

### Reviewer output quality is climbing

Story 8-5's review caught a P3 that two other reviews would likely have missed: "Final song is never auto-marked on playlist exhaustion." The fix — a terminal sweep with a new `includeCurrent: true` flag in `runCasualModeSweep`, invoked from `advanceToNext` before `songs:exhausted` — was surgical. 8-3 similarly caught `countdownRemainingMs: 0` sneaking into `session:connect` and causing a "0s" flash on the ticker. These are edge cases the spec didn't mention and unit tests wouldn't have exercised. Review is finding bugs that planning missed, which is what review is for.

### Test growth was proportional to scope, not inflated

+41 tests across five stories (+2 / +2 / +14 / +8 / +15 approximate breakdown). The big tests-per-story counts came from 8-3 (continuous mode has 10 endpoint variants: 401, 404, 403, 503, 400, enable, disable-while-counting, etc.) and 8-5 (7 unit + 4 WS integration tests for the sweep). No test-inflation fluff: the coverage is where the complexity is.

### `runCasualModeSweep` handles three call sites with one code path

The 8-5 helper takes `{ playerName?, isCatchUp? }` and runs either all-players (track change) or one-player (enable, reconnect, terminal sweep). The inner loop is identical in both modes. Four different callers, zero duplication, zero special-casing.

## Struggles & Lessons Learned

### Identity-by-display-name showed up again, and was deferred again

Three of the five stories hit the same root cause:
- 8-1: `WinOverlay` compares `selfName === winnerName` using raw user-entered strings.
- 8-2: Guest joining with the same name as `host_name` double-counts on both PlayerList rows.
- 8-4/8-5: Every casual-mode path keys by display name (`playerCasualModes: Map<string, boolean>`).

Each story's review findings explicitly defer the fix as "pre-existing project-wide pattern" — the same deferral text, three times. The pattern is that the pattern is deferred. At some point this needs a dedicated story, not another "deferred pre-existing" note. Epic 9's "Game Over Rethink" doesn't touch it either.

**Lesson:** When a deferred-work item is deferred in *three consecutive stories* with identical reasoning, it has earned its own story slot. Flag this as Epic 9 prep.

### `playerCasualModes` is new, unpersisted debt

Story 8-4 added `playerCasualModes: Map<string, boolean>` to `RoomState` but did not add it to `persistRoomState`. Story 8-5's review caught this: after a server restart, every player's casual-mode opt-in silently resets, and the catch-up sweep cannot re-emit because its target set is empty. This is a real UX regression introduced by this epic, not a carry-forward. Deferred in 8-5 review because an SQLite snapshot change would have spilled the story's scope, but it is new debt born *in* Epic 8.

### The auto-claim latch is brittle by design, and we know it

8-5's `autoClaimFired` flag only resets on `round:start`. A transient claim POST failure (network blip, 5xx, race) locks out auto-claim for the remainder of the round. The deferral reasoning is explicit and honest: "Epic 9 is about to minimize/remove the claim concept, so hardening this path would be wasted work." This is the right call, but the deferral is betting on Epic 9 shipping soon — if Epic 9 slips, we carry a known-fragile flow.

### `AudioPreset` default flipped mid-story under review pressure (8-1)

8-1 shipped with `'hype'` as the default audio preset everywhere. Review pushed to flip the default to `'minimal'` across server fallback, rehydrate, client `gameState` initial, `RoundConfigOverlay` initial, and `WinOverlay` prop default — plus all test fixtures that used `'hype'` as a placeholder. The change was correct, but it hit seven files and many tests. Had the preset default been discussed in the spec, this would have been one decision in one file. The spec said `'hype' = default` without being pressure-tested; review caught the "actually, friends-use app wants the opposite" instinct too late to avoid the sprawl.

**Lesson:** When a spec says "default X", ask *once* during planning whether the friends-use context agrees with the spec default. The cost of asking is one question; the cost of flipping it during review is one diff per call site.

### Circular module dependency introduced intentionally in 8-5

`ws.ts` imports `runCasualModeSweep` from `rooms.ts`, while `rooms.ts` already imports values from `ws.ts`. The 8-5 completion notes flag this as safe because "both sides only reference each other at call-time (not during module init), matching the existing pattern for `roomSockets`/`broadcast`." True today, but this is the first time the project has formalised a circular between its two biggest server files. Node ESM will not rescue us if a future module-init-time call sneaks in. Not a blocker, but worth watching.

## Review Feedback Patterns

Across 8-1 through 8-5: **~22 patches applied**, **~14 items deferred**, **1 dismissed** (a reviewer false-positive duplicate class in 8-4 that didn't exist in real source).

Dominant patch themes:
1. **Reconnect / rehydrate gap-closing** — 8-1 `rehydrateRooms` audioPreset default, 8-2 `hasStats` gate on reconnect, 8-4 host `casualModePlayers` unconditional seed, 8-4 guest `casualModeOn` seed, 8-5 `hasSeenRoundStart` flag for reconnect casual state. Every story shipped with a reconnect-seed bug. The seeding chain is reliable; the *activation* of seeded state at connection time keeps tripping.
2. **Unknown-enum fallback** — 8-1 `WinOverlay` `{:else}` catch-all for unknown preset, 8-3 `durationMs` type guard (deferred), 8-4 non-boolean `player:casual-mode-changed` rejection. Pattern: new enums/typed payloads need explicit `{:else}` branches and server-side validation, not `'hype' | 'deadpan' | 'minimal'` type trust.
3. **Missing negative endpoint tests** — 8-3 missing 401/404/403 tests on `POST /round/dismiss-win`, 8-3 missing `!currentRound` 409 test, 8-5 missing AC #13 reset test. Pattern: auth/ownership negative tests on new endpoints are systematically under-specced.
4. **Toast / countdown lifecycle** — 8-5 catch-up toast lingering into next round, 8-3 `countdownRemainingMs: 0` flash. Any transient UI state needs an explicit `round:start`-branch cleanup.

Dominant deferred themes (new to Epic 8):
1. **Identity-by-display-name** (3 stories touched it, 3 deferrals)
2. **`playerCasualModes` not persisted across restart** (new debt from 8-4, deferred in 8-5)
3. **Auto-claim latch on failure** (8-5, deferred on Epic 9 bet)
4. **Server-side `allowCasualMode` permission enforcement** (8-4, friends-only rationale)
5. **Duplicate countdown ticker in Host/Room pages** (8-3, DRY deferral)

## Key Takeaways

1. **Closing a four-epic deferral on day one sets the tone for the whole epic.** The `handlePlayPause` fix was the smallest commit of the epic and did the most for team morale. Treat retro action items as Story 0.
2. **The `session:connect` seeding chain is a mature, reusable architecture.** Four stories extended it with zero architectural debate. Celebrate the pattern; keep using it.
3. **Refactors that serve only the current story still pay forward.** 8-3's `startRound` extraction was for continuous mode, but 8-4 and 8-5 both landed on it. Refactor when the second caller arrives — and expect the third to arrive faster than you think.
4. **Deferring the same debt three times in one epic is a signal, not a strategy.** Identity-by-display-name has earned its own story. Schedule it, or decide explicitly to live with it permanently.
5. **Review is catching bugs planning misses, reliably.** The 8-5 terminal-sweep fix, the 8-3 `countdownRemainingMs: 0` flash, the 8-4 reconnect `casualModeOn` clobber — none were in the spec. Reviewer quality is the reason Epic 8's post-ship fix tail was small.
