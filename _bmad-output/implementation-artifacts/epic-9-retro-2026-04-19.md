## Epic Summary

A three-story epic whose final shape was nothing like its starting shape. Scope expanded mid-epic (9-2 added on 2026-04-19) and then contracted violently (9-3 deleted Continuous Mode wholesale the same day) — all before 9-2 reached dev.

1. **9-1 — Game Over Page State & Auto-Bingo.** Replaced `WinOverlay` modal with `GameOverView.svelte` as a top-level page branch. New `POST /round/next-round` with host + guest-winner auth; `BingoCard` gained a `mode` prop (active / gameover-winner / gameover-loser-their / gameover-loser-your) with honest-card "missed" shading on the loser's own view. +31 tests.
2. **9-2 — Live Round Settings & Pre-Round Simplification.** Extracted the pre-round overlay's four pill controls into a shared `AdvancedSettings.svelte` used in both the pre-round `<details>` block and the live Host Controls panel. Tests dropped 408 → 382; amended 9-2's story file inline to strip Autoplay scope before 9-2 went to dev.

All three stories landed in a single day (2026-04-19), with 9-3 designed and executed before 9-2 even started dev. Commit order: 9-1 → 9-3 → 9-2.

## What Went Well

### Epic 8 action items #2 and #3 closed inside Story 9-1, with teeth

The Epic 8 retro prep notes gave 9-1 three direct instructions. Two of them — replace `autoClaimFired`'s permanent latch (action #2), converge Casual-Mode auto-mark and manual-mark onto a single `hasBingo` effect (action #3) — landed in the same commit as the Game Over page mode. The latch rework lives inside `handleBingoClick` itself: `hasAutoClaimedThisRound` sets to `true` only after `res.status === 200`, resets on non-200 / thrown error, and resets on `round:start`. A transient network blip no longer bricks the round. The convergence is a single unconditional `$effect(() => { if (game.hasBingo) game.handleBingoClick() })` on both pages — one code path, both flows. This is the second epic running where carry-forward action items were resolved *inside story work*, not as trailing fixups. The Epic 8 pattern held.

### 9-3 was the right call, at the right moment, executed without hand-wringing

Continuous Mode as a persistent flag was a piece of state whose only effect was branching a CTA. 9-3 saw that, decided the choice belonged at the Game Over moment (two buttons, no state), and followed through with a net-deletion story: one endpoint deleted, one auth shim deleted, four WS events deleted, one countdown timer deleted, three client-state fields deleted, the CTA matrix collapsed from 6 cases to 2. Test count dropped (408 → 382) because the coverage wasn't needed anymore — no defensive bloat, no "keep the tests for safety." The decision happened the same day as 9-1 shipped (before a single real-world user had tapped the CTA matrix in anger), which is the ideal moment: 9-1's CTA code was still warm, the matrix was documented, and 9-2 hadn't started so its scope could be trimmed cleanly. This is the cheapest possible moment to delete a feature, and the team took it.

### Story file amendment as a real tool, not a hypothetical

9-3's AC #32 explicitly amends 9-2's story file. The amendment landed before 9-2 dev started: Autoplay row struck through, `HostMiniPlayer` Loop-removal task moved to 9-3, scope-amendment banner at the top of the 9-2 file. Strikethrough-with-traceability (not deletion) means future readers see the original scope and the reason it shrank. This is the first time in the project that a story has been edited mid-epic to absorb a sibling story's discovery, and it worked.

### `AdvancedSettings.svelte` as one component in two modes is a clean reuse

9-2's `<AdvancedSettings mode="pre-round" | "live" />` renders the same four pill rows in two callers: inside `<details>Advanced settings</details>` on the pre-round overlay, and inside the Round Settings section of `HostControlsOverlay` during an active round. Pre-round mode writes to component-local state; live mode fires PATCH per row with optimistic update + seq-based latest-wins guard + "Saved — applies to next song" pill. One component contract, zero duplication, identical look. The shared-component pattern is now the project's default instinct for UI that appears in both configuration and live editing — and `InfoTooltip.svelte` (also new in 9-2, touch-aware on first `pointerdown`) slots in alongside it cleanly.

### Review feedback kept finding the right class of bug

- 9-1 review caught the `role="img"` tiles nested inside `role="grid"` ARIA violation — unlikely to surface in a friends-only app but a real a11y correctness gap, fixed by switching to `role="gridcell"` with aria-label.
- 9-1 review caught `handleStartNextRound` on `HostRoomPage` not clearing `nextRoundError` on `round:start` while `RoomPage` did — an inconsistency review is uniquely good at spotting.
- 9-2 review caught a PATCH TOCTOU (re-check `currentRound?.active === true` after `await ctx.req.json()`) that neither planning nor unit tests would have caught.
- 9-2 review produced the casual-mode-revoke-and-restore design overrule that defines 9-2's final semantics — more below, but the point is the reviewer pressed on a Dev Note ("changing the permission doesn't retroactively revoke players") and surfaced a better design than the spec had.
- 9-3 review caught `/round/next-round` silent-fail (`startContinuousRound`'s four void early-returns plus unchecked `startRound`) that would have returned HTTP 200 on every failure mode. Propagating `{ ok, message? }` from `startContinuousRound` wires the transient error line to actual failures.

### Test scaling was honest — up when logic grew, down when logic shrank

+31 tests for 9-1's auto-claim convergence and Game Over rendering. +26 tests for 9-2's PATCH, hostPrefs, and AdvancedSettings flows. −26 tests in 9-3 (net) because the Continuous Mode endpoints, countdown describes, and dismiss-win cases were excised. The test suite shrank with the codebase and nobody insisted on retaining coverage for code that no longer existed. Good instinct.

## Struggles & Lessons Learned

### The spec was wrong in Story 9-2, and review had to rewrite the semantics

9-2 shipped its spec with an explicit Dev Note: *"Changing `allowCasualMode` mid-round does not retroactively revoke player toggles already on — per Story 8-4 semantics, `playerCasualModes` is separate from the room-level permission."* Under review, the Blind Hunter (or equivalent reviewer) pushed back: if the host revokes permission, the observable UX is that casual-mode players keep getting free marks until end-of-round, contradicting what the host just asked for. The fix was not small — it required introducing `RoomState.priorCasualModes?: Set<string>` for revoke/restore snapshot, broadcasting `player:casual-mode-changed {enabled:false}` per affected player on revoke, clearing `autoMarkedTileIndices` per player on revoke, running a catch-up sweep per player on restore, and adding new WS guards to reject `{enabled:true}` messages while the room permission is false. Plus it required **semantically overruling Story 8-4's per-round reset behaviour** — `playerCasualModes` now persists across round boundaries (explicit toggle-only), which is a behaviour change to a prior-epic decision. All fine and all correct — but the spec had it backwards, and the review caught it after the PATCH endpoint was already implemented.

**Lesson:** Dev Notes that read "X does not do Y" on a feature a human will actually exercise are a yellow flag. Ask during planning: "if X happens and Y doesn't happen, what does the user see?" If the answer is "the feature silently keeps doing something they just turned off," the Dev Note is wrong, not correct-by-design.

### Identity-by-display-name deferred for the fourth consecutive epic

Epic 8 retro action item #1 flagged identity-by-display-name as "has earned its own story slot." Epic 9 opened with 9-1's Dev Notes explicitly acknowledging: *"Loser-variant self-detection uses `selfName === winData.winnerName` — same bug `WinOverlay` has today. Out of scope."* 9-3 then added a small *good* to the pile — `/round/next-round` moved behind `requireAuth`, so the guest-winner `playerName === round.winnerName` spoofing path is no longer callable — but the project-wide comparison is still everywhere (claim, casual mode, win detection, tile marking). Four epics of deferral with identical reasoning. Epic 10 (if there is one) either carries this explicitly as its Story 0 or we record a formal "we will not fix this" decision. No third option.

**Lesson (unchanged from Epic 8):** When the same debt item is deferred with identical rationale across consecutive epics, the rationale is no longer a justification — it's a label. Either schedule it or mark it permanently out of scope.

### 9-1's server-side `round.winnerName` capture was speculative debt that 9-3 had to remove

9-1 added `RoundState.winnerName?: string` to the server type, set it on claim before the `round:win` broadcast, persisted it through `persistRoomState` + `rehydrateRooms`, and used it to authorize the guest-winner caller on `POST /round/next-round`. 9-3 then deleted all of it — the field, the capture, the persistence, the auth branch. It was correct work for the CTA matrix as 9-1 defined it, and it was obsolete code less than 24 hours later. 9-3's cleanup also had to cover the rehydrate edge: pre-9-3 persisted snapshots still carry `winnerName` on disk, so `rehydrateRooms` spreads it into a field that no longer exists on the type. The review fix (explicit `delete rehydrated.winnerName` post-spread) is harmless but indicative: type-level cleanups do not always ripple to on-disk state.

**Lesson:** When a feature needs "X to power Y," and X is a piece of new server state, ask once: "is there a version of Y that doesn't need X?" 9-3 proved there was — replace the guest CTA with "host-only two buttons." Had that question been asked in 9-1's spec review, 9-1 would have been smaller, 9-3 would have been smaller, and `RoundState.winnerName` would never have shipped.

### Scope didn't stabilise until mid-epic

9-2 was authored and marked ready-for-dev before 9-3 existed. 9-3 emerged from party-mode discussion on 2026-04-19 (the same day 9-1 merged) and *redefined* what 9-2 was. Had 9-2 been started earlier, a significant chunk of its code (Autoplay row, `HostMiniPlayer` Loop-button removal, the `continuousMode` plumbing through `AdvancedSettings`) would have been written and then thrown away. The "edit 9-2's story file to strip Autoplay" step saved real implementation work, but it only worked because 9-2 hadn't started dev yet. In a less solo-dev cadence, 9-2 would already have been underway and the save-by-story-amendment move would not have been available. The team got lucky on timing.

**Lesson:** Parallel story readiness creates the illusion of efficiency. If Story B's design depends on decisions only revealed during Story A's review, marking B as ready-for-dev before A ships is premature. Epic 9 works because of a single-dev serial cadence; don't generalise the pattern to a context where stories land concurrently.

### Not yet deployed — two epics on dev-only

Epic 8 retro flagged this (*"last deploy was Epic 6 + 7. Epic 8 runs on dev only so far"*) and Epic 9 is now in the same state. Two consecutive epics of work — Game Over rethink, auto-bingo, live round settings, Continuous Mode deletion — have never been exercised on prod infra. Tailscale-two-device manual runs are the only real-world shakeout. Anything that depends on Caddy, the prod build, or a real multi-room load hasn't been touched since before Epic 7.

**Lesson:** "Personal/friends app, ship when ready" is a valid stance, but two consecutive epics off-prod means the first prod bring-up for Epic 10 will carry the blast radius of 10 stories of changes at once. A deploy now — even before Epic 10's first story — isolates the surface area.

### Uncommitted polish at retro time (+138/−86 across five files)

At retrospective time, the working tree has pending edits in `AdvancedSettings.svelte`, `GuestWaitingRoom.svelte`, `HostControlsOverlay.svelte` (bulk of the diff — 161 lines reshuffled), `RoundConfigOverlay.svelte`, and `RoomPage.svelte`. None of it is committed. Not a problem for an Epic 9 retro per se (every story is marked done in sprint-status, tests are green as of last commit), but a retrospective that runs on an *in-flight* working tree can't assert readiness cleanly. Either commit or stash before declaring Epic 9 closed; otherwise the "what shipped in Epic 9" record is ambiguous.

## Review Feedback Patterns

Across 9-1 through 9-3: **~17 patches applied**, **~10 items deferred**, **10 dismissed as noise** (9-3 only, per the detailed count in its review findings).

Dominant patch themes:

1. **Error-state clearing on state transitions** — 9-1 `round:start` clearing `nextRoundError` on `HostRoomPage`; 9-3 `handleChangeItUp` clearing `nextRoundError` + `nextRoundErrorTimer` when opening the overlay. Pattern: any transient error banner needs a lifecycle teardown at every state-transition boundary, not only on its own timer.
2. **Silent-failure propagation** — 9-3 `/round/next-round` silent-fail, 9-2 broadcast payload validation at the client. Pattern: helpers that early-return `void` on failure need to propagate `{ ok, message? }` so HTTP handlers can surface real status codes, and broadcast receivers need to validate incoming payloads even when the server "shouldn't" send invalid ones.
3. **Optional-callback guarding** — 9-3 `onLetItRide?.()` / `onChangeItUp?.()` were typed optional but invoked unguarded. Pattern: `?.()` at call sites on typed-optional callbacks, or type them required.
4. **Snapshot-format drift across type deletions** — 9-3 `rehydrateRooms` still spreading `winnerName` from old snapshots. Pattern: when a field is deleted from a type that's written to disk, explicit `delete` or `pick` on rehydrate, otherwise the type no longer represents what's in memory.
5. **TOCTOU on PATCH/POST handlers** — 9-2 `currentRound?.active === true` re-check after `await ctx.req.json()`. Pattern: any state guard must be re-evaluated after every `await` boundary inside a handler.
6. **Modal layering and focus** — 9-3 disable Let It Ride / `inert` the `GameOverView` when `RoundConfigOverlay` is open behind-modal. Pattern: modals mounted atop a page that still has actionable buttons need a focus-trap or `inert` on the underlying content.

Deferred themes:

1. **Identity-by-display-name** — 9-1 loser-variant `selfName === winnerName` (4th consecutive deferral).
2. **Reconnect replays `round:win`** — 9-1 Dev Note, 9-3 review re-raise. Reconnecting mid-Game-Over loses access to Let It Ride.
3. **Buffered `round:end` during Game Over** — 9-3 review, exposed more by deleting `round:dismissed`.
4. **401 re-auth prompt UX** — 9-3 review, pre-existing across every host endpoint.
5. **`session:connect` wire-protocol versioning** — 9-3 review, no project-wide version field exists.

## Key Takeaways

1. **Closing carry-forward action items inside the story they apply to works, repeatedly.** Epic 8 did it with the `handlePlayPause` fix. Epic 9 did it with the auto-claim latch and the Casual Mode / auto-bingo convergence. Story 0 of each epic should continue to be "whatever the last retro flagged." Two epics is a pattern now; keep it going.

2. **Deleting a feature the same day it ships is the cheapest possible moment.** 9-3 caught Continuous Mode before a single player had tapped the CTA matrix in real play, before 9-2 had started dev, and before the 10-second countdown timer had any field evidence. The window for "delete the state, not move it" closes fast — a week later, 9-3 would have had prod users, a dev sprint on 9-2 to rewind, and a "but we already built it" argument. The lesson: when a reviewer says "this piece of state exists only to branch a CTA," take the deletion seriously *now*, not after the CTA is hardened.

3. **Story file amendments are a real tool in a solo cadence.** 9-3 editing 9-2's ACs (strikethrough with traceability) saved dev work that would otherwise have been thrown away. The pattern is safe only because 9-2 hadn't started dev; in a concurrent team, the amendment would have been a rebase-after-the-fact. The takeaway is not "always use story amendments" — it's "parallel story readiness is an illusion of efficiency." Don't mark Story B ready-for-dev when Story A's review can reshape B's design.

4. **Specs that carry Dev Notes of the form "X does not do Y" on user-exercised behaviours are yellow flags.** 9-2's "changing `allowCasualMode` mid-round does not retroactively revoke players" was literally correct per the 8-4 contract, and literally wrong for what a host expects. Review caught it, at the cost of rewriting the semantics and introducing a new `priorCasualModes` snapshot structure. The preventive question at planning: "if X happens and Y doesn't happen, what does the user see?"

5. **Deferring the same debt four times in a row isn't deferral anymore — it's a label.** Identity-by-display-name has now been "deferred — project-wide pre-existing pattern" in Epics 3, 4, 5, 6, 7, 8, and 9. Epic 10 either schedules it or formally accepts it. No fifth iteration.

6. **Net-deletion stories shrink the test suite honestly.** 9-3 dropped 408 → 382 by deleting tests that covered code that no longer exists. No one insisted on keeping coverage "for safety." Test count as a vanity metric would punish this; the project doesn't track it that way.

## Correction (post-retro, 2026-04-19)

The identity-by-display-name framing in this retro (and in Epic 8's, and in 9-1's Dev Notes) overstates the problem. The server-side 4009 `'name taken'` check at [ws.ts:382-387](src/server/ws.ts#L382-L387) prevents two guests with the same name from being connected simultaneously, so the "two Alices both see the winner variant" failure mode is not actually reachable. The 4009 guard closes the interesting `selfName === winnerName` collision entirely.

Residual identity holes are narrower than the retro implied:

- **Guest name == host name** — 4009 checks only `roomState.guests`, not `room.host_name`. Real bug, originally flagged in Epic 8 retro under Story 8-2 (PlayerList double-counts).
- **Disconnect → someone else claims the name → original reconnects** — slot has been overwritten; original player's card/state is gone.
- **Rehydrate after server restart** — cards keyed by name; a different human typing the same name post-restart inherits the prior card.

None of these are "two of the same person on screen at once" bugs, and none have surfaced in friends-only play. Action item #3's "fourth consecutive deferral / force an Epic 10 decision" framing is therefore retracted: the debt isn't worth a dedicated story at this scale. The two narrow holes above can be addressed opportunistically if either ever bites.
